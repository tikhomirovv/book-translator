# PRD

## Executive Summary

**Проблема:** Перевести целую книгу через LLM нельзя одним запросом — контекст ограничен, качество падает без памяти о терминах и сюжете, процесс долгий и может оборваться.

**Решение:** CLI, который извлекает текст из файла, режет на смысловые чанки (параграфы), переводит последовательно с накоплением контекста, сохраняет прогресс по идентификатору перевода и выдаёт Markdown.

**Критерии успеха MVP:**
- PDF успешно извлекается в текст с сохранением параграфов.
- Перевод можно прервать и продолжить командой с `translation-id`.
- В выходном Markdown есть frontmatter (язык назначения, дата, модель, оценка стоимости/токенов).
- Качество перевода субъективно приемлемо на 1–2 тестовых книгах для личного использования.
- Unit- и integration-тесты проходят без реальных вызовов внешних API.

## Текущий scope (MVP)

### В scope

| Область | Описание |
|--------|----------|
| Вход | PDF |
| Выход | Markdown + YAML frontmatter |
| Интерфейс | CLI: флаги и интерактивный режим |
| Языки | Только целевой язык (`--to`); исходный определяет модель |
| Allowed languages | Список разрешённых языков в конфиге; новые языки включаются после проверки качества |
| LLM | OpenAI-compatible API; несколько провайдеров через одну абстракцию |
| Прогресс | Файловое хранилище без БД; идентификатор перевода |
| Контекст | Гибрид: глоссарий + summary + overlap; стратегия через абстракцию (MVP: фиксированное окно) |
| Надёжность | Retry с exponential backoff; сохранение каждого успешного чанка |
| Rate limit | Простая задержка между запросами (таймаут из конфига) |
| Промпты | ID промпта (`nonfiction`, `fiction`, …) в одном конфиге |
| Логи | Уровни debug/info/warn/error; progress bar в CLI |
| Тесты | Unit + integration с mock LLM |

### Вне scope MVP (non-goals)

- Веб-интерфейс (загрузка/скачивание файлов).
- База данных (PostgreSQL и т.п.) — только файлы.
- EPUB, MOBI, DOCX на входе и выходе.
- Параллельный перевод **разных** чанков (контекст между чанками последовательный).
- Автоматическая оценка качества перевода (benchmark).
- Биллинг, multi-tenancy, очереди задач.

## Пользовательские сценарии

### US-1: Новый перевод

**Как** пользователь, **я хочу** запустить перевод PDF на выбранный язык, **чтобы** получить Markdown.

**Критерии приёмки:**
- Команда с флагами: `--input`, `--output`, `--to`, `--prompt-type` (опционально).
- Интерактивный режим: те же параметры через вопросы в консоли.
- Создаётся `translation-id` (UUID v4); прогресс пишется на диск.
- Исходный язык не передаётся — модель определяет сама.
- Целевой язык должен быть в `allowed_languages` конфига.

**Пример (флаги):**
```bash
translator translate \
  --input book.pdf \
  --output book.ru.md \
  --to ru \
  --prompt-type nonfiction
```

### US-2: Продолжить прерванный перевод

**Как** пользователь, **я хочу** указать `translation-id`, **чтобы** продолжить с последнего успешного чанка.

**Критерии приёмки:**
- Команда `resume --id <translation-id>`.
- Загружается состояние: последний чанк, глоссарий, summary, метаданные.
- Уже переведённые чанки не переводятся повторно.

### US-3: Статус перевода

**Как** пользователь, **я хочу** видеть прогресс, **чтобы** понимать, сколько осталось и сколько потрачено.

**Критерии приёмки:**
- Команда `status --id <translation-id>`.
- Показывает: N/M чанков, токены/стоимость (если доступно от API), время, ошибки последнего запуска.

### US-4: Список активных переводов

**Как** пользователь, **я хочу** видеть список переводов, **чтобы** выбрать нужный `translation-id`.

**Критерии приёмки:**
- Команда `list` — краткая таблица: id, исходный файл, целевой язык, прогресс, дата.

## Требования

### Извлечение и чанкинг

- Текст из PDF извлекается через порт `TextExtractor` (MVP: `razvandimescu/gopdf`, без LLM).
- Постобработка: reflow переносов строк, нормализация параграфов.
- Чанк = группа из **10 параграфов** по умолчанию (`chunk.size_paragraphs` в конфиге).
- Скользящее окно: последние 2 параграфа предыдущего чанка включаются в overlap следующего.
- Команда `extract` — только извлечение текста в файл (отладка без LLM).

### Управление контекстом

- **Инкрементальное извлечение:** каждый чанк обрабатывается переводом и параллельным обновлением контекста.
- **LLM-managed memory (MVP):** контекстная модель получает предыдущий контекст + новый фрагмент и возвращает пересобранный блок памяти.
- **Лимит контекста:** `llm.context.max_tokens` — и API-лимит ответа, и указание в промпте; клиент не обрезает контекст сам.
- **Перевод чанка N** использует контекст из чанков `< N`.
- **Позже:** альтернативные стратегии через `ContextManager` (например `contextwindow`).

### Параллелизм

- **Между чанками:** строго последовательно (контекст зависит от предыдущих).
- **Внутри чанка:** перевод + обновление контекста параллельно (`errgroup`).

### LLM и промпты

- Провайдеры: OpenAI-compatible (OpenAI, Anthropic через gateway, LM Studio и т.д.).
- Два независимых профиля в конфиге: `llm.translation.*` и `llm.context.*` (model, temperature, max_tokens).
- Параметры модели — в **конфиге**, не в `.env`. API key опционален для локальных OpenAI-compatible серверов.
- Промпты по ID в том же конфиге: `nonfiction`, `fiction`, …
- Типы промптов на чанк: `system`, `translation`, `context_extraction`.
- **Полный перевод:** модель не должна опускать части книги (оглавление, списки глав, заголовки — это содержание). «Очистка» — только технические артефакты PDF (переносы, кодировка), без удаления текста. Без комментариев от модели.

### Конфигурация

**`.env` (секреты и техника, не в git):**
- API keys, base URL провайдера.
- `LOG_LEVEL` (опционально).

**`config.yaml` (в git, дефолты):**
- `chunk.size_paragraphs`, `chunk.overlap_paragraphs`
- `llm.translation.*`, `llm.context.*`
- `translation.paragraph_from`, `translation.paragraph_to` (опционально, для тестов)
- `request_delay_ms`
- `allowed_languages`
- `prompts` (по ID)

**`config.local.yaml` (не в git):** перезаписывает любые ключи дефолтного конфига (паттерн Viper: defaults → config.yaml → config.local.yaml → env).

### Хранение прогресса (без БД)

Корень: `translations/<translation-id>/`

| Файл | Назначение |
|------|------------|
| `state.json` | Прогресс, последний успешный чанк, глоссарий, summary, метаданные |
| `chunks/` | Переведённые фрагменты по чанкам |
| `source.meta.json` | Путь к исходнику, хеш, целевой язык, prompt-type |

`translation-id` — UUID или короткий hash; передаётся в `resume` и `status`.

### Выходной файл

- Markdown с YAML frontmatter: целевой язык, дата, модель, провайдер, оценка токенов/стоимости, `translation-id`.
- Тело — склеенный перевод чанков.

### Учёт токенов и стоимости

- Брать `usage` из ответа OpenAI-compatible API, если провайдер возвращает.
- Если usage нет — WARN в лог, нули в `state.json`; локальный подсчёт через **tiktoken** — после MVP.
- Накопление: `state.json` + frontmatter выходного Markdown.

### Логирование

- Структурированные логи (zerolog или аналог).
- Уровни: DEBUG, INFO, WARN, ERROR.
- Progress bar совместим с логами.
- В будущем — экспорт в Grafana (не в MVP).

### Ошибки

- Retry: exponential backoff, до 3 попыток на запрос.
- После успеха чанка — немедленная запись на диск.
- При фатальной ошибке — понятное сообщение + сохранённый прогресс для resume.

## Ограничения

- Качество зависит от модели и промпта; нет формальной метрики в MVP.
- Стоимость перевода книги может быть высокой (зависит от модели и длины).
- PDF со сложной вёрсткой/сканами может плохо извлекаться (OCR — вне MVP).
- Фиксированный лимит `llm.context.max_tokens` может терять детали на очень длинных книгах (модель сама сжимает память).

## AI System Requirements

### Инструменты

- OpenAI-compatible Chat Completions API.
- Опционально (позже): отдельная «дешёвая» модель для summarization контекста.

### Оценка качества (MVP)

- Ручная проверка 1–2 книг.
- Чеклист: термины согласованы, имена не «прыгают», стиль ровный, нет обрывов на границах чанков.

### Оценка качества (после MVP)

- Golden set из фрагментов + human review.
- Опционально: LLM-as-judge на выборке.
